# なぜ TQC なのか

> この文書は「読んで理解して、レポート④(学習の取り組み)に自分の言葉で書き直す」ための説明資料です。ここの文章をそのままレポートにコピペしないでください(採点時に生成AI・コピペのセルフチェックが行われます。README §2.5 参照)。

## 1. TQC とは何か

**TQC(Truncated Quantile Critics)** は、**SAC(Soft Actor-Critic)** という強化学習アルゴリズムを土台にした派生版です。SB3 本体(`stable_baselines3`)ではなく、拡張パッケージの **sb3-contrib** に入っています(`from sb3_contrib import TQC`)。

用語がわからない場合は README [§11 用語集](../README.md#11-用語集)も参照してください。

## 2. なぜ授業の基本である PPO ではなく TQC なのか

一番大きい理由は、**このチームが CPU・限られた時間で回している**という制約に対して、TQC の方が相性が良いからです。

- **PPO はオンポリシー** … 集めた経験(「こう動いたらこの報酬だった」というデータ)を1回使ったら捨てて、また新しく経験を集め直します。データの使い方としては「もったいない」方式です。
- **TQC(SAC系)はオフポリシー** … 経験を「リプレイバッファ」という倉庫に溜めておき、何度も繰り返し使い回して学習します。同じ経験から何度も学べるので、**同じ試行回数でもより多くを学べる**(サンプル効率が良い)のが特徴です。

CPU 学習・Kaggle のセッション時間上限(約12時間)という限られたリソースの中で完走方策を目指す今回の課題では、この「経験を使い回せる」性質が実務上の決め手になります。

## 3. TQC の強み(仕組みの要点)

TQC が SAC よりさらに安定して学習できる理由は、大きく3つの工夫にあります。

**① SAC譲りの「ほどよいランダムさ」を保つ探索**
方策が完全に一本道に固まってしまわず、常に少しランダムに動く余地(エントロピー)を残しながら学習します。これにより、局所的な失敗パターンに固執せず、色々な歩き方を試し続けられます。

**② 報酬の「予測」を1つの数字ではなく分布で持つ**
多くの強化学習アルゴリズムは「この行動をしたら将来もらえる報酬の合計はだいたい○○」と1つの数字で予測します。しかしこの予測は外れることがあり、外れた予測を信じて突き進むと学習が不安定になります(いわゆる「自分の見積もりを過信して失敗する」問題)。TQC はこの予測を1つの数字ではなく、**「悲観的な見積もり〜楽観的な見積もりまでの分布」**として持ちます。

**③ 分布の甘すぎる部分を切り捨てる(truncate)+複数の批評家(critic)で見積もる**
たとえるなら、1人の審査員ではなく複数の審査員(critic のアンサンブル)に採点してもらい、さらに**極端に甘い点数だけを切り捨てて**平均を取るイメージです。これにより「実際より良く見えているだけの行動」を過大評価しにくくなり、結果として学習全体が安定します。名前の「Truncated(切り捨てられた)」はこの工夫から来ています。

まとめると、TQC は「経験を使い回せる(サンプル効率)」+「自分の予測を過信しない(過大評価の抑制)」の両方を備えたアルゴリズムで、限られた試行回数で安定した完走方策を目指す今回の課題に向いている、というのが採用理由です。

## 4. このリポジトリでの位置づけ

- TQC は SB3 公式アルゴリズム集ではなく、拡張パッケージ **sb3-contrib** が提供しています。
- 連続制御タスク(BipedalWalker のような「関節をなめらかに動かす」系のタスク)のベンチマークで上位に来ることが多いアルゴリズムです。
- `src/train.py` では、YAML に書いたキーが TQC の引数名と一致していれば自動で TQC に渡る仕組み(`TQC_PARAM_NAMES`)になっています。新しいハイパラを試したいときの編集場所は README [§4.5 実験の地図](../README.md#45-実験の地図何を変えたいときどこを編集するか) の表Aを参照してください。
